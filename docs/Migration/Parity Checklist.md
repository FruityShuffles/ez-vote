# Parity Checklist

The behaviors the React app must preserve to reach parity with the frozen Flutter app. Most entries were learned from a production incident (a closed bug) or are a deliberate design decision тАФ exactly the subtle, easy-to-lose behavior that a rewrite silently regresses. This is the **test plan** for each Phase-2 surface port (M8тАУM16): when you port a surface, every box in its section must be verified before that surface is considered at parity.

## How to use this

- Entries are grouped by the surface that implements them, headed with the migration issue that ports that surface.
- Each entry: a stable **ID**, the observable behavior, its **source** (closed issue `#N` or a decision/architecture doc), and a **Verify** note тАФ how to prove the React build preserves it.
- **Observable behavior, not mechanism.** Many bugs were Flutter/Riverpod-specific (provider invalidation, `Timer.periodic` races). React will use different mechanisms (Supabase realtime subscriptions, TanStack Query, etc.). What must survive is the *observable behavior and the edge case the incident exposed* тАФ re-test the scenario, don't port the implementation.
- Check a box (`- [ ]` тЖТ `- [x]`) when the React surface is verified against that behavior, ideally with an automated test or a recorded side-by-side run against Flutter (M18).
- Tabulation invariants (┬з1) are already machine-guarded by the **M2 golden corpus**; client-side derivation (┬з4, marked тЯР) is guarded by **M23**. Those sections cross-reference their guard rather than duplicating it.
- **Holistic sign-off** of this checklist is driven by [[Migration/Cutover Plan]] (M17): M18 walks the cutover plan's test-election matrix side-by-side against Flutter and ticks these boxes; the cutover gate is "every ЁЯФ┤ box checked, every ЁЯЯб checked or with a recorded divergence."

Legend: ЁЯФ┤ high-risk (core voting correctness or a hard-won incident) ┬╖ ЁЯЯб behavioral but lower-stakes ┬╖ ЁЯОи visual/UX polish that still counts as parity.

## 9. Realtime тАФ M15

These were Flutter polling bugs; the React port **keeps polling** (a 10s `useElectionRealtime` hook, mechanism-identical to Flutter) rather than adopting Supabase Realtime channels, which would need a backend publication change; the channel migration is tracked as a post-cutover follow-up. **Re-test the scenario, not the timer logic** тАФ the observable guarantees are what matter.

- [x] ЁЯФ┤ **RTM-01** тАФ A change that occurs in the **first interval after the screen mounts** is not missed: results, ballot count, voters list, and public-ballots list all refresh for it (the #82 "first tick seeds but doesn't invalidate" trap). _Source:_ #82. _Verify:_ have another user submit a ballot within seconds of opening the detail screen тЖТ everything updates. _(M15: `useElectionRealtime` keeps `lastResultsUpdatedAt` null at mount тАФ never seeds тАФ so the first detected timestamp invalidates; covered in `useElectionRealtime.test.ts`. Public-ballots list isn't a ported surface yet.)_
- [x] ЁЯФ┤ **RTM-02** тАФ Concurrent refreshes don't race: overlapping updates can't overwrite the change-tracking baseline and drop an update (the #66 in-flight guard). _Source:_ #66. _Verify:_ throttle the network, fire rapid changes тЖТ no dropped update. _(M15: `inFlight` ref guard mirrors Flutter's `_polling`; an in-flight tick is skipped, not queued тАФ tested.)_
- [x] ЁЯФ┤ **RTM-03** тАФ Realtime recovers from a transient error instead of dying silently: after a fetch/subscription error, it re-establishes rather than staying dead until navigation. _Source:_ #67. _Verify:_ induce a transient failure тЖТ updates resume without leaving the screen. _(M15: an errored detail query is invalidated to retry; a thrown poll fetch is swallowed with the baseline left intact and the interval kept running, so the next tick recovers тАФ both tested.)_
